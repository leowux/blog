## 导语

今天最值得写的 AI 技术话题是：Open-source memory for coding agents, synced over SSH。

这个话题在 2026 年 7 月 16 日北京时间凌晨进入 Hacker News 前排；本次写作时，HN 页面显示约 10 分和 6 条讨论。热度不在于分数绝对值，而在于它抓住了编码代理正在变成日常工具后的一个痛点：代理已经在本机留下了大量调试、设计和决策记录，但下一次会话通常仍然从零开始。

## 核心观点

deja-vu 的思路很直接：不要先构建一个新的记忆平台，而是索引 Claude Code、Codex 和 opencode 已经写到本地磁盘的会话历史。它提供命令行搜索、MCP `recall` 工具、会话启动时的自动上下文注入，以及通过 SSH 或共享目录在多台机器之间同步记忆。

这个选择很有代表性。过去很多 agent memory 方案默认走向云端、向量库和专门的捕获 API；deja-vu 反过来强调本地倒排索引、零模型调用、凭据脱敏和可检查的文本检索。它牺牲了一部分语义搜索的想象空间，换来的是更低的延迟、更清晰的数据边界，以及对既有会话日志的 retroactive 索引。

HN 讨论里也有人直接问它和其他记忆工具的差异。作者强调的不是“记得更多”，而是组合：零依赖、纯文本检索、索引阶段脱敏、SSH 点对点同步、以及新会话启动时自动喂入少量相关上下文。这说明社区对 agent memory 的要求正在从概念演示转向工程可控性。

## 技术影响

第一，编码代理的记忆层会越来越像开发工具链的一部分，而不是聊天产品的附属功能。团队真正需要的是“这类 bug 三周前怎么修过”“这个项目的迁移约束是什么”“上次为什么放弃某个方案”这类可复用上下文。把这些信息沉淀在本地索引中，可以让不同 harness 之间共享经验，而不是被锁在某个单一会话窗口里。

第二，本地优先会成为 agent memory 的重要分支。代理日志往往包含代码片段、内部路径、失败命令、凭据痕迹和业务上下文；如果记忆系统默认上传或不可审计，团队很难放心扩展。deja-vu 在索引和分享阶段做 secret redaction，并声明索引与搜索没有网络路径，这类设计会成为企业采用编码代理时的基本门槛。

第三，简单检索仍然有位置。向量搜索适合模糊语义，但工程排障经常依赖精确词、错误码、文件名、函数名和命令片段。一个毫秒级、可解释、可脚本化的 lexical memory 层，可能比“更聪明但更黑盒”的记忆系统更容易被纳入日常开发流程。

## 工程启发

如果团队已经在使用编码代理，可以先从三个问题入手。

第一，盘点代理到底写了哪些本地日志。Claude Code、Codex、opencode、aider 或自研 harness 的会话数据如果各自孤立，就很难形成组织记忆。

第二，把记忆当作敏感数据处理。索引前脱敏、分享前再脱敏、明确同步边界，应该和 agent sandbox、网络 allow-list 一样成为默认要求。

第三，不要只追求“长期记忆”，还要追求“可引用的短记忆”。很多高价值上下文只需要在当前项目、当前分支或当前故障周期内被找回；过度抽象成全局知识库，反而会增加噪声和治理成本。

deja-vu 的意义不在于它已经解决了所有 agent memory 问题，而在于它把问题拉回了工程现场：历史会话已经存在，先让代理能可靠、私密、可检查地用起来。
