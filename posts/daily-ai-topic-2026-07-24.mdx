## 导语

今天值得关注的 AI 工程话题是：Harness 工程不够：为什么软件工厂会失败。

HumanLayer 的一篇新文章在 Hacker News 获得了 300 余分和 220 余条讨论。它针对近来流行的“软件工厂”叙事提出反论：给编码 Agent 配好上下文、工具、沙箱和自动审查，确实能提高实现速度；但这不等于团队已经解决了长期维护、需求判断和生产质量的问题。这个观点尤其值得与近期关于 Harness Engineering 的讨论放在一起看：harness 很重要，但不能把它当作取消工程责任的理由。

## 核心观点

文章把传统交付流程拆成需求、实现、代码审查、测试、发布、监控和用户反馈等环节。Agent 首先压缩的是“实现”时间；若随后再用 Agent 加速 review 和回归测试，系统仍然面对同一个难题：谁来判断需求是否正确、变更是否符合架构意图、以及累积的局部修改是否正在削弱可维护性。把人工 review 一笔勾销，只是把成本转移到事故、返工和更难定位的系统性问题上。

这是作者的工程判断，并非已经被单篇研究证明的定律。文章也明确承认，模型在一次性任务和小型新项目上的表现已有明显进步；它质疑的是模型能否在数月乃至更长的真实代码库演进中，持续改善而不是侵蚀系统结构。当前基准大多奖励一次任务的正确性，很少直接衡量长期可维护性，因此不能用 leaderboard 分数替代这项验证。

## 技术影响

Faros 发布的 2026 年遥测报告为这场讨论提供了值得检验的信号。该报告覆盖 2.2 万名开发者、4,000 个团队的两年数据；其页面显示，高 AI 采用与交付吞吐提升同时出现，但也伴随更大的 PR、更多每 PR 缺陷、更长 review 时间、更高的每 PR 事故数和更高代码 churn。报告把这些结果描述为统计相关性，而不是“AI 导致事故”的因果证明；组织、项目类型和采用方式都可能影响结果。即便如此，它足以提醒团队：用产出量或合并数单独衡量 Agent 成功，会漏掉质量债务。

工程上的关键转变是把“速度”与“可逆性”一起设计。对低风险、边界清楚的任务，Agent 可以承担较高的自主度；对跨模块重构、权限、支付、数据迁移和生产变更，则需要更强的规格、可执行验收条件、分阶段发布和明确的人类 owner。自动化应优先生成可复查的证据，而不是只生成更多代码：测试结果、运行前后指标、迁移计划、回滚步骤和架构决策都应成为交付物。

## 工程启发

1. 不要把“无人 review”设为成熟度目标。按风险设置 review 门槛：小改动可让 Agent 预审并抽查，高影响改动仍由清楚业务和架构背景的人确认。
2. 将质量信号前置到任务入口。任务说明应包含受影响边界、不可破坏的约束和验收命令；让 Agent 在写代码前先展示计划、风险与验证路径。
3. 同时记录领先和滞后指标。除交付量、耗时和成功率外，持续观察返工率、代码 churn、review 时长、缺陷逃逸率、事故数和回滚次数，并按服务与任务风险分层。
4. 给自动化保留熔断与回退。渐进发布、feature flag、可回滚迁移和清晰 owner，能让团队从一次错误中恢复；它们比再增加一个提示词或循环更接近生产可靠性。

“软件工厂会失败”不应被理解为停止使用编码 Agent，而应理解为拒绝把自动化速度误当成系统能力。Harness 是让 Agent 获得正确上下文和执行边界的基础；而长期可靠性仍取决于团队是否能持续定义目标、验证结果，并在错误发生前后掌握反馈回路。
